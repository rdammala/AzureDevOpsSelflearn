CI/CD Pipelines: Complete Overview

Interview-Ready & Learning Guide

Quick Summary: Two-tier pipeline system: NonOfficial (auto CI on master) + Official (manual production deployment with code signing).

1. Pipeline Architecture Overview

1. Restore NuGet packages
2. Compile code (warnings-as-errors)

4. Publish artifacts

4. Interview-Ready Answers

Q: "Explain your two-tier pipeline system"

Answer:

NonOfficial Pipeline: Runs automatically on every master commit. Builds, tests, and deploys to dev and staging. No code signing required. Gives fast feedback to developers.

Official Pipeline: Manual trigger, signs all binaries with certificates, requires explicit approval before production deployment. Ensures production code is authorized and auditable.

Why two pipelines? Separates fast developer feedback from slow, audited production releases. Dev/staging get new code immediately. Production requires ceremony and approval.

Q: "What happens if tests fail?"

Answer:

Pipeline halts. No deployment happens. Old version stays live (safe). Developer gets notification, reviews logs, fixes code, commits, and build auto-retriggers.

The safety: broken code never reaches users because failed builds block deployment.

Q: "How do you avoid deployment failures?"

Answer:

Multiple gates:

  • Build gate: Compile + unit tests
  • Integration gate: Emulator-based integration tests
  • Functional gate: Full app test against staging environment
  • Approval gate: Manual approval before prod (Official pipeline only)

Each gate is a checkpoint. Only proceed if previous gate passes.

Infrastructure as Code (Bicep)

Interview-Ready & Learning Guide

Quick Summary: Bicep is a DSL that compiles to ARM templates, providing type safety, reusability, and clean syntax for Azure infrastructure.

1. Why Bicep Over ARM Templates?

Bicep provides:

  • Clean Syntax - ~30 lines vs 60+ lines for ARM JSON
  • Type Safety - Errors at compile time, not deploy time
  • Native String Interpolation - ${var} instead of concat()
  • Modules - Reusable components (not complex linkedTemplates)
  • Easy Diffs - Git diffs are readable

Example: A simple Web App is ~30 lines in Bicep vs ~60 lines in ARM JSON.

2. Bicep Syntax Basics

param location string
param environment string
param instanceCount int = 1

var appServicePlanName = 'plan-notifications-${environment}'

resource appServicePlan 'Microsoft.Web/serverfarms@2021-02-01' = {
  name: appServicePlanName
  location: location
  sku: {
    name: 'B1'
    capacity: instanceCount
  }
}

output appServicePlanId string = appServicePlan.id

3. Modules: Reusability

Why? Define infrastructure once, use everywhere. Consistency across all domains.

Module Example:

module frontendApp './modules/webapp.bicep' = {
  name: 'frontendDeployment'
  params: {
    name: 'app-chat-frontend-${environment}'
    location: location
    appServicePlanId: appServicePlan.id
  }
}

4. Two-Level Deployment

Level 1: envType (subscription-level, shared expensive resources)
├─ Cosmos DB (shared by dev + staging)
├─ Key Vault (shared)
└─ Cost: ~$50/month

Level 2: env (resource-group-level, per-environment sizing)
├─ Dev App Service: B1 (~$10)
├─ Staging App Service: B2 (~$50)
└─ Prod App Service: P1V2 (~$500)

5. Interview-Ready Answer

Q: "Why use Bicep instead of ARM templates?"

Bicep gives clean syntax vs JSON nightmare, type safety (errors at compile time), native string interpolation, modules for reuse, and easy git diffs. A simple Web App is 30 lines Bicep vs 60 lines JSON. Multiply across 50+ domains = huge maintenance saving.

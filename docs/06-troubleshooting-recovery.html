Troubleshooting & Recovery

Debugging Deployment Failures

Quick Summary: Common deployment failures and recovery procedures: Build fails → Fix → Tests fail → Fix → Swap fails → Rollback/Fix.

Build Stage Troubleshooting

Build Failed: Compilation Error

error CS0103: Name 'logger' does not exist

Solution:
1. Run locally: dotnet build Chat/Chat.slnx
2. Find: logger not imported
3. Add: using Microsoft.Extensions.Logging;
4. Commit: git commit -am "Fix: add missing using"
5. Push: git push origin fix/bug-name
6. Build auto-retriggers

Build Failed: NuGet Restore Error

NU1101: Unable to find package "SomePackage"

Investigation:
1. Check package name spelling
2. Verify in Directory.Packages.props
3. Check NuGet feed access (nuget.config)

Solution: Fix package version or contact NuGet admin

Build Failed: Test Failure

Fix either the code or the test, depending on whether behavior changed intentionally.

Deploy Stage: Bicep Failure

Syntax Error

Bicep compilation error: Expected parameter, found 'resource'

Solution:
1. Run locally: bicep lint template.bicep
2. Check indentation, brackets
3. Fix syntax
4. Deploy via pipeline

Resource Quota Exceeded

The resource quota of 800 has been reached

Solution:
1. Count current resources
2. Delete unused resources
3. Request quota increase from Azure
4. Re-queue deployment

Deploy Stage: Functional Tests Failure

API Returns 500

Solution:
1. Check app logs: az webapp log tail -n app-name
2. Look for exceptions
3. Fix locally
4. Commit, push, build auto-retriggers

Connection Timeout

Solution:
1. Check if app startup is slow
2. Verify connection strings
3. Check firewall rules
4. Increase timeout if needed

Production: Post-Swap Monitoring

Error Rate Spike Detected

IMMEDIATE ACTION: ROLLBACK

az webapp deployment slot swap \
  -n app-notifications-frontend-prod \
  -g rg-notifications-frontend-prod \
  --slot staging

Result: v1.0 back live in <1 second
Error rate drops immediately

Interview Answer

Q: "How do you handle production failures?"

We have multiple safety nets. Build fails → pipeline stops. Tests fail → pipeline stops. Functional tests fail → swap blocked. If issues slip past to production, we instantly rollback the slot swap. Old version is back live in < 1 second. Then team investigates, fixes, and re-deploys when ready. The key: assume things will go wrong and design for rapid recovery.

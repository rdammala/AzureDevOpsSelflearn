Containers & Orchestration

Docker, Kubernetes, and Infrastructure Automation

Quick Summary: Docker containerizes applications (image → container), Kubernetes orchestrates at scale (deployments, replicas, load balancing), Ansible manages infrastructure as code.

Docker Basics

Image vs Container

Image:      Template (blueprint) - like a class
Container:  Running instance    - like an object

Example:
Image:      Windows + .NET 8 + Chat.Frontend.dll
Container:  Running Chat.Frontend listening on port 80

Building an Image

FROM mcr.microsoft.com/dotnet/aspnet:8.0
WORKDIR /app
COPY bin/Release/net8.0/publish/ .
ENTRYPOINT ["dotnet", "Chat.Frontend.dll"]

Build: docker build -t chat-frontend:1.0 .
Run:   docker run -p 80:80 chat-frontend:1.0

Kubernetes Orchestration

The Problem: Running 100+ containers across 50 servers manually = impossible.

The Solution: Kubernetes automates scaling, load balancing, rolling updates.

Key Kubernetes Concepts

  • Pod - Smallest unit (usually 1 container)
  • Deployment - Define how many replicas of pod to run
  • Service - Load balancer + DNS for pods
  • ConfigMap - External config, not hardcoded
  • Secret - Passwords, keys, encrypted

Deployment Example

apiVersion: apps/v1
kind: Deployment
metadata:
  name: chat-frontend
spec:
  replicas: 3
  selector:
    matchLabels:
      app: chat-frontend
  template:
    metadata:
      labels:
        app: chat-frontend
    spec:
      containers:
      - name: frontend
        image: chat-frontend:1.0
        ports:
        - containerPort: 80

Ansible for Infrastructure

Purpose: Automation across many servers without agents.

---
- hosts: webservers
  tasks:
  - name: Install Git
    apt:
      name: git
      state: present
  
  - name: Clone repository
    git:
      repo: https://github.com/example/repo.git
      dest: /app/repo

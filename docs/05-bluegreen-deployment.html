Blue-Green Deployments

Zero-Downtime Deployment Strategy

Quick Summary: Slot-based blue-green deployment: Two app slots (staging + production), test on staging, swap traffic with zero downtime, instant rollback capability.

The Problem: Downtime

Traditional Approach:
  Production Running (v1.0)
  ↓
  Stop Application (⏹️ 5 minutes downtime)
  ↓
  Deploy New Code (v2.0)
  ↓
  Start Application
  ↓
  ⚠️  Customers see service unavailable!

The Solution: Slot Swap

BEFORE SWAP:
Production: v1.0 (live) → Serving 1000+ requests/sec
Staging: [empty]

DEPLOY:
Production: v1.0 (live) → Serving requests
Staging: v2.0 (testing) → Running tests

TESTS PASS:
Swap production ← staging

AFTER SWAP:
Production: v2.0 (live) → Serving requests (no interruption!)
Staging: v1.0 (backup) → 1 swap away for rollback

Why This Works

  • ✅ Zero Downtime - Traffic never stops
  • ✅ Instant Rollback - Swap back in <1 second if issues
  • ✅ Safe Testing - Tests run before traffic switches

Rollback Scenario

Error spike detected in production after swap
  ↓
INSTANT ROLLBACK:
  az webapp deployment slot swap ...
  ↓
v1.0 BACK LIVE (< 1 second)
Error rate drops immediately

Interview Answer

Q: "How do you do zero-downtime deployments?"

We use slot-based blue-green deployments. We deploy new code to a staging slot, run functional tests, and if they pass, swap staging and production slots. Traffic switches instantly. Old version stays on the other slot for instant rollback if issues arise. The swap is < 1 second.

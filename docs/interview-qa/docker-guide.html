📌 Section 1: Docker Concepts

Docker packages applications and dependencies into containers—lightweight, portable units that run consistently across environments.

Docker Components:
✓ Image: Blueprint for containers
✓ Container: Running instance of an image
✓ Registry: Repository for images
✓ Dockerfile: Instructions to build images
✓ Compose: Multi-container orchestration

📌 Section 2: Dockerfile Structure

FROM python:3.11-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install -r requirements.txt

COPY . .

EXPOSE 8080

CMD ["python", "app.py"]
Multi-stage builds: Use multiple FROM statements to reduce final image size and improve security.

📌 Section 3: Building and Running Containers

# Build image
docker build -t myapp:1.0 .

# Run container
docker run -p 8080:8080 --name mycontainer myapp:1.0

# List running containers
docker ps

# Stop container
docker stop mycontainer

# Remove image
docker rmi myapp:1.0

# Push to registry
docker tag myapp:1.0 myregistry.azurecr.io/myapp:1.0
docker push myregistry.azurecr.io/myapp:1.0

📌 Section 4: Docker Compose

version: '3.8'

services:
web:
build: .
ports:
- "8080:8080"
environment:
- DB_HOST=database
depends_on:
- database

database:
image: postgres:13
environment:
- POSTGRES_PASSWORD=secret
volumes:
- pgdata:/var/lib/postgresql/data

volumes:
pgdata:
Security: Never hardcode secrets in Compose files. Use .env files or secrets management.

📌 Section 5: Container Registries

  1. Docker Hub: Public registry for sharing images
  2. Azure Container Registry (ACR): Private registry for enterprise
  3. Image Tagging: Use semantic versioning (v1.0.0)
  4. Image Scanning: Check for vulnerabilities
Best Practice: Always tag images with version numbers. Use 'latest' tag carefully!

📌 Section 6: Networking and Storage

# Create network
docker network create mynetwork

# Run container on network
docker run --network mynetwork --name web myapp
docker run --network mynetwork --name db postgres

# Create volume
docker volume create myvolume

# Run with volume
docker run -v myvolume:/data myapp
Volumes vs Bind Mounts: Use named volumes for production, bind mounts for development.

📌 Section 7: Best Practices

  • ✓ Keep images small: Use minimal base images
  • ✓ Security: Scan images for vulnerabilities
  • ✓ Don't run as root inside containers
  • ✓ Use health checks to detect failures
  • ✓ Log to stdout/stderr for container visibility

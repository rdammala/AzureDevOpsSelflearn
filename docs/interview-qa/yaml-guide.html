📌 Section 1: Introduction to YAML Pipelines

YAML (YAML Ain't Markup Language) is a human-readable data serialization format used to define CI/CD pipelines. Modern DevOps teams use YAML pipelines for infrastructure-as-code approaches.

Why YAML?
✓ Version controlled with code
✓ Human-readable and maintainable
✓ Supports complex branching logic
✓ Reusable templates and variables

📌 Section 2: Basic YAML Structure

A YAML pipeline has 5 core components:

  1. Trigger: When the pipeline runs
  2. Pool: Where jobs execute (agent)
  3. Variables: Reusable values
  4. Stages: Logical groupings of jobs
  5. Jobs: Units of work with steps
trigger:
- main

pool:
vmImage: 'ubuntu-latest'

variables:
buildConfiguration: 'Release'

stages:
- stage: Build
jobs:
- job: CompileCode
steps:
- script: echo Building...

📌 Section 4: Variables and Secrets

variables:
version: '1.0.0'
buildPlatform: 'Any CPU'
global_var: 'shared-value'

stages:
- stage: Build
variables:
stage_var: 'build-specific'
jobs:
- job: Test
variables:
job_var: 'job-specific'
Secret Variables: Use Azure DevOps variable groups for passwords, API keys, tokens. Never commit secrets!

📌 Section 5: Conditions and Logic

jobs:
- job: BuildJob
condition: eq(variables['Build.SourceBranch'], 'refs/heads/main')
steps:
- script: echo Building from main branch

- job: TestJob
dependsOn: BuildJob
condition: succeeded()
steps:
- script: echo Running tests after build
Common Mistake: Using succeeded() for conditional execution. Remember: succeeded() only checks previous job, not all dependencies.

📌 Section 6: Best Practices

  1. Use templates: Reduce duplication with YAML templates
  2. Secure secrets: Store sensitive data in variable groups
  3. Fail fast: Run quick tests before expensive operations
  4. Cache dependencies: Speed up builds with caching
  5. Parallel execution: Run independent jobs simultaneously
Template Example: Create reusable job templates in /templates folder for consistency.

📌 Key Takeaways

  • ✓ YAML pipelines are version-controlled and repeatable
  • ✓ Stages run sequentially; jobs within stages can run parallel
  • ✓ Use conditions to control execution flow
  • ✓ Always protect secrets with variable groups
  • ✓ Test your pipeline YAML syntax before committing
